Metric Definitions

Prompt Precision Score (Based on First Pass Yield - FPY)

To provide a real-world baseline for our "Prompt Precision" metric, we've adapted the industry-standard concept of First Pass Yield (FPY), commonly used in manufacturing and software engineering.

Definition: Prompt Precision measures the percentage of initial prompts that generate code which is functionally correct, adheres to specified architectural constraints, and is integrated with only minor, non-functional edits (e.g., renaming variables, adding comments). It quantifies the AI's efficiency and its ability to understand and execute complex requirements correctly on the first attempt.

Formula:

Prompt Precision = (Successful First-Try Prompts / Total Prompts) * 100

Example: For a feature requiring 20 prompts, if 19 produced usable code on the first try, the score is (19 / 20) * 100 = 95.

Project Instruction Set

Core Architectural Philosophy

All development is guided by a three-perspective model to ensure security, efficiency, and responsiveness:

  • Server-Authoritative Control: The server is the single source of truth for all game-critical systems, including scoring, economy, and core game state. This prevents cheating and ensures data integrity.
  • Player-Centric Feedback: The owning client (the player) handles non-critical, performance-intensive tasks locally. This includes rendering complex visual effects and processing input to provide immediate, responsive feedback.
  • Observer Broadcasting: Other clients receive only the essential information needed to represent the player's actions, such as synchronized animations and client-side visual effects. This minimizes network traffic and ensures a smooth experience for all observers.

Networking Best Practices

To optimize network performance, we adhere to a principle of event bundling:

  • Efficient Synchronization: A single network event, such as a `SyncVar` change or an `RPC` call, should be used to trigger multiple related local effects. For example, setting an `IsAttacking` boolean to `true` should not only sync the state but also trigger local animations, sound effects, and UI updates on observing clients.

FishNet Implementation Standards

Consistent implementation of FishNet components is critical for stability and performance:

  • NetworkObject: The foundational component for any object synchronized over the network.
  • NetworkTransform: Mandatory for any networked object that moves. It automatically handles position and rotation syncing and can be configured for either server or client authority.
  • NetworkAnimator: Used for objects with an Animator component. It provides granular control over which animation states are synchronized, allowing us to exclude non-essential animations to conserve bandwidth.
  • Physics & Time Management: All physics and time-sensitive transformations must use the `TimeManager` class, leveraging `TickDelta` for calculations to ensure perfect synchronization with the server's tick rate.
  • SyncVar Declaration: All `SyncVar` fields must be declared as private and read-only to enforce proper encapsulation, like so: private readonly SyncVar<T> _variableName = new SyncVar<T>();
  • Lifecycle Constraints: The `IsOwner` property is not accessed within the `Awake()` method, as network ownership is not established at that stage of the object lifecycle.

Data Persistence

Prompt:

Task: Allow players to place custom paintings (decals) in the world that persist across server restarts.

Method: When a player places a decal via `PaintingUIManager`, a ServerRpc sends the texture data to `DecalSpawner`, which calls `PersistentDecalManager`. NPCs Supported

Prompt:

Task: Spawn and manage a large number of NPCs without impacting server performance, ensuring they respawn correctly.

Method: The `NPCSpawner` reads spawn points from a JSON file and groups them into clusters. It spawns NPCs in clusters up to a `_maxNPCs` limit. When an NPC dies, `OnNPCDespawned` is called, adding the NPC's cluster point to a respawn queue. A timed server process, `ProcessRespawnQueue`, periodically respawns NPCs from this queue, ensuring the world remains populated. The system uses FishNet's object pooling for efficient instantiation.

Test Method Prompt:

Prompt: Configure the server to spawn a high number of NPCs (e.g., 1000) and monitor server-side performance metrics, specifically framerate and CPU load. The test scenario should involve clients eliminating numerous NPCs in different clusters. The test will log the respawn queue's behavior and timing to ensure NPCs are correctly re-added to the world over time, without exceeding the maximum population cap.

Outcome:

The cluster-based spawning and queued respawn system allows for thousands of NPCs to be managed efficiently. The auto-healing nature of the respawn queue guarantees world population density remains consistent, providing a scalable and robust solution for NPC management.

Prompt:

Task: Player disconnects must not cause data loss or leave orphaned objects in the world.

Method: The `SessionManager` listens for the `OnRemoteConnectionState` event. When a client disconnects (`RemoteConnectionState.Stopped`), the `OnClientDisconnected` method is triggered. We can identify which types of features are best suited for AI-driven development and which may require more hands-on senior developer oversight, optimizing our development pipeline.
